# 01_3d_Topo_Check

This folder is a topology-check case for the 3D SUBOFF H-O grid. It is mainly
used before replacing the dummy OpenFOAM `constant/polyMesh/points` file with
coordinates generated by the MATLAB H-O grid generator. The goal is to verify
that point ordering, boundary-face ordering, cell ownership, patch names, and
decomposition behavior are compatible with the later real SUBOFF coordinates.

The case uses a simple dummy H-O block layout:

- four O-type blocks around the body, corresponding to the cylindrical part of
  the grid,
- one square H-type block in the center,
- separate `INLET1`, `INLET2`, `HULL`, and `OUTLET` patches.

## Workflow Idea

OpenFOAM topology files such as `faces`, `owner`, `neighbour`, and `boundary`
are generated from the dummy `system/blockMeshDict`. The MATLAB script under
`constant/polyMesh_yuanshi/` writes a replacement `points` file using the
intended SUBOFF H-O point ordering. This folder lets users check that those two
views of the mesh agree before using the generated coordinates in a production
case.

The key idea is:

```text
dummy OpenFOAM topology + generated SUBOFF points ordering = valid H-O mesh
```

If the generated `points` ordering does not match the OpenFOAM topology, the
mesh can have inverted cells, wrong patch faces, or broken boundary data even
when the coordinate values themselves look correct.

## Quick Start

Run commands from `01_3d_Topo_Check` unless stated otherwise.

1. Generate the dummy topology.

   ```bash
   blockMesh
   ```

   This creates `constant/polyMesh/faces`, `owner`, `neighbour`, `boundary`,
   and a dummy `points` file from `system/blockMeshDict`.

2. Inspect the dummy H-O layout.

   The important block parameters are:

   ```text
   NJ 120
   NK 128
   NL NK/4
   NI 40
   ```

   `blockMeshDict` contains five blocks:

   - four O-type/cylindrical blocks with dimensions `($NI $NJ $NL)`,
   - one square/H-type block with dimensions `($NI $NL $NL)`.

3. Generate the SUBOFF point ordering with MATLAB.

   From `constant/polyMesh_yuanshi`, run:

   ```matlab
   grid
   ```

   This writes a `points` file and Tecplot inspection outputs. The generated
   `points` file should only replace OpenFOAM's dummy `constant/polyMesh/points`
   after its ordering has been checked against the dummy topology.

4. Test topology selection and refinement.

   `topoSetDict` creates a `cellSet` named `refineCells`:

   ```text
   box (-11 -3 -3) (1 3 3)
   ```

   `refineMeshDict` then refines this set using:

   ```text
   useHexTopology yes
   geometricCut   no
   ```

5. Test decomposition.

   `system/decomposeParDict` is set for 16 subdomains:

   ```text
   numberOfSubdomains 16
   method scotch
   ```

   The commented `simple` coefficients use:

   ```text
   n (1 4 4)
   ```

   This case can be used to check whether the generated topology and patch
   ordering survive decomposition and reconstruction.

6. Use `run.sh` only when you are ready to overwrite generated case state.

   The script runs `foamCleanPolyMesh`, restores `0.orig` and
   `constant/polyMesh.orig`, decomposes, runs `topoSet`, refines, reconstructs,
   and launches solver tests. It is useful for a full topology workflow, but it
   modifies processor folders and mesh state.

## Main Topology Files

| File | Role |
| --- | --- |
| `system/blockMeshDict` | Defines the dummy five-block H-O topology and patch names. |
| `constant/polyMesh/faces` | OpenFOAM face connectivity generated by `blockMesh`. |
| `constant/polyMesh/owner` | Owner-cell list for each face. |
| `constant/polyMesh/neighbour` | Neighbour-cell list for internal faces. |
| `constant/polyMesh/boundary` | Patch names, face counts, and patch start indices. |
| `constant/polyMesh/points` | Dummy point coordinates initially; later replaced by the generated SUBOFF coordinates. |
| `constant/polyMesh_yuanshi/grid.m` | Writes the replacement point coordinates using the intended H-O ordering. |

## Point Ordering

`constant/polyMesh_yuanshi/grid.m` writes points in two major blocks.

First, it writes the O-type/cylindrical surface layers:

```matlab
for K = 1:(NL*4)
    for J = 1:(NJ+1)
        for I = NI+1:-1:1
            ...
        end
    end
end
```

Then it writes the inner square/H-type block, excluding the outer square edge
already covered by the O-grid connection:

```matlab
for J = NL:-1:2
    for K = 2:NL
        for I = NI+1:-1:1
            ...
        end
    end
end
```

The generated point count is:

```matlab
P_NUM = (NI+1)*(NJ+1)*NK + (NL-1)^2*(NI+1)
```

This formula is one of the main invariants. If `NI`, `NJ`, `NK`, or `NL`
changes, both the dummy topology and generated points must be regenerated
together.

## Patch Ordering

The dummy topology defines these patches:

| Patch | Meaning |
| --- | --- |
| `INLET1` | Square/H-type inlet face. |
| `INLET2` | O-type inlet ring, split across four block faces. |
| `HULL` | Body wall and square-to-O connection surface. |
| `OUTLET` | Downstream O-type outlet faces. |

Keep these patch names stable because later folders use `INLET1`, `INLET2`,
and `HULL` explicitly for inlet point extraction, boundary data, and solver
setup.

## Decomposition And Reconstruction Checks

The test workflow can be used to check whether cell and face ordering remain
valid through parallel operations:

```bash
redistributePar -decompose -cellDist -parallel -overwrite
topoSet -parallel
refineMesh -parallel -overwrite
redistributePar -reconstruct -parallel -overwrite -constant
```

After reconstruction, inspect:

- `constant/polyMesh/boundary` for patch counts and start-face indices,
- `constant/polyMesh/owner` and `neighbour` for consistency,
- `checkMesh` output for negative volumes or bad faces,
- decomposed `processor*/constant/polyMesh` folders if testing parallel
  topology behavior.

## Notes And Limitations

- This folder is for topology and ordering checks, not for final production
  coordinates or final solver physics.
- Do not replace only `constant/polyMesh/points` unless the point count and
  ordering match `faces`, `owner`, `neighbour`, and `boundary`.
- `run.sh` is operational and destructive: it cleans mesh state, restores
  `0.orig` and `constant/polyMesh.orig`, overwrites decomposition, reconstructs
  meshes, and removes selected processor time folders.
- If `NK` changes, keep `NL=NK/4` and preserve the four-sector O-grid ordering.
- If block dimensions or patch definitions change, regenerate and re-check all
  downstream point extraction and inlet boundary workflows.
